# 🛡️ GenAI Dashboard V2 — 登入權限調整與 IIS 發佈指南

本檔案為專案維護與系統管理員使用手冊。本系統在正式 IIS 部署與內網運作下，採用**「純 Windows 自動登入與後端動態授權」**架構。所有的權限策略、登入模式切換以及測試模擬，皆可透過修改單一設定檔案完成，**完全不需更動 C# 程式碼或重新編譯**。

---

## 📌 一、 修改文件與架構原則

### 1. 修改哪份文件？
* 主機上的設定核心為根目錄的 **`appsettings.json`**（以及 `appsettings.Production.json` / `appsettings.Development.json`）。
* 所有的驗證設定都集中在 JSON 的 **`"Auth"`** 區塊中。

### 2. 唯讀 JSON 架構承諾
* **`web.config` 已設定為唯讀 `appsettings.json`**：系統在 IIS 上運行時（`ASPNETCORE_ENVIRONMENT = Production`），已移除所有 XML `Auth__` 變數覆寫，確保後端 100% 只聽從主機上 `appsettings.json` 的設定，維護管理最直覺。

---

## 🔐 二、 四大核心情境與配置實作 (`appsettings.json`)

請打開或修改 `appsettings.json` 中 `"Auth"` 區塊的對應欄位：

### 情境 1：如何改為「真實 Windows 桌機自動登入」？ (正式環境推薦)
當系統發佈到公司內網正式上線時，我們希望系統自動偵測並驗證當前員工桌機的真實 Windows 工號，且不再顯示任何手動輸入密碼介面：

```json
{
  "Auth": {
    // ⭐️ 留空字串 ""：強制系統透過 IIS Negotiate 取回當前訪問使用者的真實 Windows 工號
    "SimulatedWindowsAccount": "",

    // ⭐️ 關閉手動登入：隱藏登入頁面的手動輸入帳號密碼分頁
    "AllowManualLogin": false,

    // ⭐️ 關閉 LDAP/測試帳號：確保純內網安全無縫驗證
    "TestAccounts": { "Enabled": false, "Accounts": [] },
    "Ldap": { "Enabled": false }
  }
}
```

---

### 情境 2：如何「模擬特定帳號登入」？ (開發或主管授權測試專用)
當工程師在本機開發，或系統管理員在正式主機想臨時以特定主管/同仁的工號視角檢查權限與畫面，無需他們本人親自到場輸入帳密：

```json
{
  "Auth": {
    // ⭐️ 填入您要模擬的目標工號 (例如 "00058897" 或 "yu-tinglin")
    //   不論實際操作者的 Windows 帳號是誰，後端會一律將目前連線視為此工號並給予對應權限
    "SimulatedWindowsAccount": "00058897",

    "AllowManualLogin": false
  }
}
```
> 💡 **測試結束後**：只需將 `"SimulatedWindowsAccount": ""` 改回空字串，或將檔案存檔，後端會立即無縫還原回真實 Windows 帳號，不用重啟伺服器！

---

### 情境 3：策略 A — 允許「全廠一般使用者皆可瀏覽」 (開放訪客模式)
如果您想要的是「全廠任何人，只要連進這台 IIS 伺服器，就算此前未曾在他帳號裡設定過，都能進來觀看儀表板」：

```json
{
  "Auth": {
    "SimulatedWindowsAccount": "",

    // ⭐️ 設為 true：當訪客進站且資料庫 Accounts 表中查無此工號時，
    //   後端會自動幫他建立一筆 RoleLevel = "user" 的新帳號，並賦予預設選單權限
    "AutoProvisionWindowsAccounts": true,
    "DefaultRoleIds": [ "role_1" ]
  }
}
```

---

### 情境 4：策略 B — 只允許「已在『帳號管理』頁面建檔的使用者」瀏覽 (門禁安全管控模式)
如果您的企業規範是「嚴格門禁」：只有被系統管理員在後台**「帳號管理」**中建立過授權的同仁才得以登入；外人或陌生工號一律擋在門外：

```json
{
  "Auth": {
    "SimulatedWindowsAccount": "",

    // ⭐️ 設為 false：關閉自動註冊功能。
    //   若使用者的工號未曾在後台「帳號管理」建檔，系統立刻拒絕登入，
    //   並於前台顯示「[工號] 無瀏覽此網頁的權限」
    "AutoProvisionWindowsAccounts": false
  }
}
```

---

## 🚀 三、 遠端 IIS 發佈排錯指南：如何排除「狂跳主機帳號密碼輸入對話框」？

很多時候把網站部署到遠端 IIS 後，用瀏覽器訪問時沒有自動帶入本機桌機帳號，反而彈出一個「要您輸入遠端伺服器 Windows 帳號密碼」的視窗。

這**絕對不是**專案程式碼的 Bug，而是因為 **IIS 驗證模式搭配不當** 或是 **瀏覽器安全區域 (Intranet Zone) 未放行** 所致。請依序確認以下兩大設定：

### 步驟 1：檢查遠端 IIS 管理員 (`IIS Manager`) 驗證設定面板
打開伺服器上的 `IIS 管理員` ➡️ 點選本站台 (`GenAI`) ➡️ 雙擊右側的 **「驗證 (Authentication)」** 圖示，確認狀態必須長這樣：

| 驗證項目 | 狀態設定 | 為什麼要這樣設？ |
| :--- | :--- | :--- |
| **匿名驗證 (Anonymous Authentication)** | 🟢 **啟用 (Enabled)** | **必開！** 允許 HTML、CSS、JS 與首頁初次載入，並讓後端靜默探測狀態。 |
| **Windows 驗證 (Windows Authentication)** | 🟢 **啟用 (Enabled)** | **必開！** 讓 `/api/Auth/WhoAmI` 能夠向 IIS 取得訪問者的 Windows 桌機帳號。 |
| **基本驗證 (Basic Authentication)** | 🔴 **停用 (Disabled)** | **必關！** 若開啟此項，瀏覽器只要遇到權限檢查一定會跳出醜陋的帳密對話框。 |
| **表單/摘要驗證 (Forms / Digest)** | 🔴 **停用 (Disabled)** | 保持停用，避免驗證通訊協定衝突。 |

> 🔧 **進階提供者排序檢查**：
> 於「Windows 驗證」點選右鍵 ➡️ **「提供者 (Providers)」**，確認列表中第一順位為 **`Negotiate`**（Kerberos 優先），第二順位為 **`NTLM`**。

---

### 步驟 2：設定瀏覽器的「近端內部網路 (Local Intranet Zone)」(最關鍵！)
為什麼 IIS 設好了 Windows 驗證，瀏覽器仍然不直接送帳號，而要彈出視窗問密碼？
因為 Edge / Chrome / IE 預設會將「不在同一個網域命名空間或以 IP/主機名訪問的網站」當成**「網際網路 (Internet Zone)」**。為了保護安全，瀏覽器**不敢隨便送出您的 Windows 桌機憑證**，因此彈窗向您再次確認密碼！

#### 👉 個人/測試電腦立即排解教學 (以 Windows 控制台為例)：
1. 點選 Windows 搜尋欄或執行 `control.exe`，打開 **「控制台」** ➡️ 點選 **「網際網路選項 (`Internet Options`)」**。
2. 切換到 **「安全性 (`Security`)」** 頁籤。
3. 點選 **「近端內部網路 (`Local Intranet`)」** 圖示 ➡️ 點選右方 **「網站 (`Sites`)」** 按鈕。
4. 在彈出視窗中勾選所有選項，並點進 **「進階 (`Advanced`)」**。
5. 將遠端 IIS 網站的網址（例如 `http://p58esiap12` 或 `http://主機名/GenAI`）輸入並點選 **「新增 (`Add`)」**。
6. 回到上一層的「安全性」頁籤（仍點選著「近端內部網路」），點選下方 **「自訂等級 (`Custom level...`)」**。
7. 滑到最底部的 **「使用者驗證 (`User Authentication`) -> 登入 (`Logon`)」** 區塊：
   * 將選項從「只在近端內部網路區域自動登入」確認改勾選為：  
     ☑️ **「自動以目前的 Windows 使用者名稱及密碼登入 (`Automatic logon with current user name and password`)」**（或維持近端內部網路自動登入）。
8. 點選「確定」保存並重新開啟瀏覽器。  
   👉 **結果**：從此打開網頁一秒無縫登入，**絕不再彈出任何輸入密碼對話框！**

#### 👉 企業大規模全廠部署最佳實踐：
如果公司內部有數千台電腦要使用此系統，請 **IT / AD 網域管理員**透過 **GPO (Group Policy 群組原則)** 執行以下統一推送：
1. 將本系統的 IIS 伺服器網址 (`http://p58esiap12` 等) 透過 GPO 加入全體員工桌機的 **「信任的近端內部網路 (`Intranet Zone`)」** 網站清單。
2. 設定 Intranet Zone 驗證策略為 **`Automatic logon with current user name and password`**。
如此一來，全廠員工不論在新舊電腦或何種 Windows 版本打開網址，皆能達成企業級的「零對話框、零密碼、一秒無縫登入」！
